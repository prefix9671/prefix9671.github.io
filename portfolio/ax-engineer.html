Case 01 · Development

PR 기반 변경관리·릴리즈 문서·Smoke Gate 통합

환경개발 조직 60명 이상, 프로젝트당 7~8명, 5개 프로젝트, 16개 제품, 릴리즈당 변경 30~40건문제SVN Merge 후 릴리즈 직전 약 3일 동안 개발자가 코드를 다시 읽어 문서를 작성성과전년 동월 Main Release 대비 변경 누락 약 99% 감소, 전달시간 약 60% 단축

공용 Trunk와 고객사별 특수 패치가 뒤섞인 환경에서, 담당 개발자의 기억에 의존하던 변경 의도를 PR 시점에 구조화하고 실제 앱 동작으로 검증했습니다.

99%변경 누락 감소
60%전달시간 단축
75%고객지원 문의 감소
1~2회분기별 DLL 문제 차단
1. PR 단위 변경 분석
PR 생성 시 Claude API 기반 Agent가 Commit, Jira, C++ 심볼, 기존 릴리즈 문서와 영향 범위를 분석합니다.
2. 실제 WinApp 재현
Rust 기반 Custom MCP가 화면 정보와 UI 상태를 CLI로 노출하고 버튼 클릭, TextBox 수정, 검사 Window 설정과 광학 검사 흐름을 실행합니다.
3. 문서·증적 생성
변경 의도, 목적, 영향 범위, 제약사항을 Docmost에 작성하고 PR 본문에 링크를 삽입합니다. 실제 Screenshot이 첨부되지 않으면 Mock 이미지가 남아 검토 누락을 드러냅니다.
4. 승인과 Gate
실행 결과와 문서가 다르면 Fail, 재현되지 않으면 NEED CHECK로 개발자에게 반환합니다. 개발자가 변경 내용을 승인하고 QA가 릴리즈 착수를 승인합니다.

운영 전환

처음부터 Merge를 막지 않았습니다. 6개월 동안 Warning-only로 운영하며 오탐과 예외를 줄인 뒤 공식 결재를 거쳐 Smoke 실패 시 Merge를 차단하는 Gate로 전환했습니다.

의도적으로 포기한 범위회귀 전용 장비를 확보할 수 없었고 기존 Simulator도 CUDA 기반 3D Image Draw 경로를 재현하지 못했습니다. 따라서 CPU 기반 기능과 재현 가능한 UI 흐름만 자동화하고 GPU·광학·실장비 의존 영역은 QA 수동 검증으로 남겼습니다.
결과: 릴리즈 직전 일괄 문서 작성에서 PR 단위 Incremental Review로 전환해 변경 누락과 고객 문의를 줄이고, 개발자가 기능 작업에 더 몰입할 수 있는 구조를 확보했습니다.
면접용 상세 메모 펼치기

1. 원래 무엇이 문제였나

  • 릴리즈 범위가 확정되는 시점은 배포 직전 약 3일 전이었고, 개발자가 SVN Merge 후 코드를 다시 읽으며 릴리즈 노트를 작성했습니다.
  • 전체 개발 조직은 60명 이상이며, 프로젝트당 7~8명이 참여했습니다. 5개 프로젝트와 16개 제품을 관리했고 Main Release마다 변경 30~40건이 포함됐습니다.
  • 알고리즘 수정, 공용 옵션, 공용 Trunk 변경처럼 간접 영향을 주는 항목이 자주 빠졌습니다.
  • 커밋 메시지는 대부분 “고객사 요청사항으로 수정” 수준의 한 줄이어서 담당 개발자 외에는 변경 의도를 알기 어려웠습니다.
  • 의도된 변경이 문서에서 누락되면 QA가 이전 버전과 달라진 결과를 신규 결함으로 등록하는 Fake Issue가 매 릴리즈마다 발생했습니다.
  • 고객사도 패치 후 결과값이 달라졌다고 항의했고, 담당 개발자에게 하루 평균 약 2건의 지원 전화가 집중됐습니다.

2. 왜 템플릿과 교육으로 해결되지 않았나

  • 릴리즈 노트 템플릿, 개발자 교육, Jira 티켓, 코드 주석, 내부 문서화 등을 지속적으로 시도했지만 변경관리는 여전히 불안정했습니다.
  • 고객사 전용 특수 패치가 Main과 장기간 분리되며 사실상 커스텀 제품처럼 변했고, Main Merge 시 Conflict가 반복됐습니다.
  • 상세 정보를 많이 제공할수록 QA 검증 범위와 개발자 대응 업무도 늘어났기 때문에 개발자에게는 최소한의 정보만 제공하는 편이 단기적으로 유리했습니다.
  • C++ 코드 복잡도가 높아 단순 Diff나 Jira만으로는 호출 관계와 실제 사용자 영향을 추론하기 어려웠습니다.

3. 실제 설계

  • PR 생성 시 Pipeline이 Claude API Agent를 자동 호출했습니다.
  • 입력은 Commit과 PR, Jira, C++ 심볼 분석, 기존 문서, Smoke 결과, UI Toolkit Dump, 실제 WinApp 실행 결과를 함께 사용했습니다.
  • Rust 기반 Custom MCP는 화면 정보 CLI 노출, 버튼 클릭, TextBox 수정, 상태 변화 반환, 검사 Window 그리기, 부품 광학 검사 기능을 제공했습니다.
  • Agent 결과는 Docmost에 작성하고 PR 본문에 링크를 삽입했으며 원본 HTML은 서버에 보관했습니다.
  • 개발자가 Docmost를 수정하면 Agent가 다시 읽어 HTML을 갱신하고, PR Merge 시 연결된 HTML도 함께 Stage에 배포했습니다.

4. AI 결과를 어떻게 통제했나

  • 초기에는 내부 함수 이름을 그대로 쓰는 문제가 있어 코드 심볼을 사용자 UI 명칭으로 변환했습니다.
  • 변경 기능은 Custom MCP로 실제 실행하고, 재현되지 않으면 CI 진입 전에 NEED CHECK로 개발자에게 반환했습니다.
  • 문서 설명과 실제 실행 결과가 다르면 Fail 처리했습니다.
  • 초안에는 일부러 Mock Screenshot을 넣어 개발자가 실제 기능을 실행하고 진짜 Screenshot으로 교체하도록 유도했습니다.
  • 최종 Merge는 개발자, 릴리즈 착수 승인은 QA가 담당했습니다.

5. 실제 누락을 잡은 사례

공용 Trunk의 그림자 연산 처리 엔진이 마이너 버전으로 업데이트된 적이 있습니다. Agent는 변경을 감지해 문서를 작성했지만 실제 Screenshot이 교체되지 않아 Mock 이미지가 그대로 남았습니다. 담당자를 추적한 결과 알고리즘 팀장이 버그 수정 후 반영 사실을 잊고 있었고, 변경 부분을 다시 확인한 뒤 정식 승인했습니다.

6. 현업 반대와 해결

“코드를 직접 본 개발자가 릴리즈 노트를 써야 가장 정확하다.”
  • 초기에는 자동 생성 문서가 기존 수기 문서보다 약 4배 길어져 검토 부담이 커졌다는 반발이 있었습니다.
  • 릴리즈 직전 30~40건을 한 번에 검토하는 대신 PR마다 새로 추가된 내용만 확인하는 Incremental Review로 바꿨습니다.
  • AI 작성을 원하지 않는 개발자는 동일 양식에 수기로 작성할 수 있게 했지만 실제 Screenshot과 Smoke Gate는 동일하게 적용했습니다.
  • 제약사항 공개에 반대하는 개발자에게는 “고객이 실패했을 때 셀프 해결할 수 있는 메시지를 제공하면 담당 개발자의 전화 수신을 줄일 수 있다”는 효과로 설득했습니다.
  • 도입 후 고객지원 문의는 하루 약 2건에서 이틀에 약 1건으로 감소했습니다.

7. 실패와 운영 한계

  • 장비를 회귀 전용으로 고정할 수 없었고 실제 테스트에 계속 사용됐습니다.
  • Simulator는 GPU 경로를 고려하지 않아 CUDA 미사용 3D Image Draw만 재현할 수 있었습니다.
  • 전체 AOI 회귀 자동화는 포기하고 CPU·UI·Smoke 범위로 한정했습니다.
  • 시스템 장애는 Claude Code가 Runbook에 따라 조치하고, 실패하면 담당자에게 Escalation하도록 구성했습니다.

8. 면접에서 강조할 한 문장

AI가 문서를 대신 쓴 것이 아니라, 담당 개발자의 기억에만 있던 변경 의도를 PR 시점에 끌어내고 실제 앱 실행으로 검증해 문서·승인·배포에 연결했습니다.

Case 02 · QA

스크린샷 기반 TC·Jira·Git 연계 자동화

문제수기 테스트 이후 TC 갱신, 증적 정리, Jira 작성, 개발 연계에 반복 시간이 발생성과QA 수동 검수·행정시간 30% 이상 단축

테스트 행위 자체를 무리하게 자동화하지 않고, 결과를 옮기고 설명하고 연결하는 주변 행정업무부터 제거했습니다.

1. 기존 행동 유지
QA는 기존처럼 테스트하고 실패 화면의 스크린샷을 저장합니다.
2. 자동 매핑
파일명과 테스트 규칙을 이용해 스크린샷을 TC와 연결하고 재현 단계와 증적 후보를 구성합니다.
3. 이슈·코드 연계
Jira 티켓 생성, 증적 첨부, 관련 Git 작업 연결을 자동 수행합니다.
4. 예외 검토
판정이 모호하거나 중복 가능성이 있는 건만 QA가 최종 검토합니다.

통제 포인트

모든 쓰기 작업을 직렬화하고, 자동 생성 결과는 최종 상신 전에 QA가 확인하도록 분리했습니다.

결과: 현업 테스트 방식의 변화 없이 검수 및 행정 처리시간을 30% 이상 단축

Case 03 · HR

비정형 전자결재 기반 휴가 관리 자동화

문제ERP 미지원 휴가를 Excel로 관리하고, 전자결재 문서는 표준 양식 없이 자유서술성과휴가 문서 관리시간 99% 감소

규칙 기반 처리가 어려울 정도로 제각각 작성된 문서를 Agent가 구조화하되, 원문과 수정 이력을 항상 남겼습니다.

1. 문서 수집
승인된 전자결재 문서를 웹에서 수집하고 기안자와 원문 링크를 식별합니다.
2. 비정형 해석
Agent가 휴가 유형, 사용일, 시간 단위, 발생 근거를 구조화합니다.
3. Excel 갱신
Excel COM 자동화로 기존 휴가 대장을 갱신해 담당자의 익숙한 업무 화면을 유지합니다.
4. 근거 보존
내부 DB에 원문 링크, 추출 결과, 수정 이력과 처리 시각을 저장합니다.

통제 포인트

문서 해석 결과가 충돌하거나 날짜·휴가 유형이 불명확한 건은 자동 반영하지 않고 담당자 검토 대상으로 분리했습니다.

결과: 휴가 문서 조회·전사·대장 갱신 반복을 제거해 담당 시간 99% 감소

Case 04 · Finance

야간근로 수당·택시비 지급 검증 자동화

문제세콤, 근태, 전자결재와 영수증을 사람이 대조해 야근식대·초과수당·택시비를 산정성과재무 반복 정산업무 80% 이상 감소

급여 관련 결과를 LLM이 확정하지 않도록 기본 산정, 택시비 보정, 최종 Validator를 분리했습니다.

1. 3중 교차검증
세콤 퇴근시간, 내부 근태시스템, 전자결재 승인 정보를 대조해 일치하는 근로자의 야간근로 시간과 지급 대상을 산정합니다.
2. Excel 반영
정상 건을 Excel COM 자동화로 지급 대장에 반영하고 원천 근거를 내부 DB에 저장합니다.
3. Taxi Agent 보정
택시비 전자결재를 제출한 임직원을 식별해 영수증과 신청 정보를 기준으로 지급 항목을 보정합니다.
4. 정적 Validator
최종 산출된 야간근로 시간, 전자결재 존재 여부, 영수증 금액과 지급 결과를 결정론적으로 재검증합니다.

통제 포인트

Agent는 문서 해석과 후보 생성에만 사용하고, 최종 금액과 지급 여부는 원천 데이터 기반 Validator가 확인합니다. 불일치 건은 지급 대장에서 분리합니다.

결과: 재무팀의 급여·복리후생 반복 정산 작업을 80% 이상 감소시키고 감사 증적 확보

AX 이력서로 돌아가기 · 회사명, 개인 식별정보, 내부 원문과 실제 데이터는 공개하지 않고 구조와 성과만 익명화했습니다.
