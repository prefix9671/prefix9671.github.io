Case 01 · Release Quality

PR 기반 변경관리·릴리즈 문서·Smoke Gate 통합

공용 Trunk, 고객사별 커스텀 패치, 한 줄 커밋 메시지가 혼재한 레거시 C++·SVN 환경에서 변경 의도를 QA 검증 범위로 전환하고, 실제 WinApp 실행 결과가 문서와 일치해야 Merge될 수 있도록 품질 게이트를 구축했습니다.

99%↓명시적 변경 누락
60%↓릴리즈 정보 전달시간
75%↓고객지원 문의
1~2회분기별 DLL 문제 차단
환경개발 조직 60명 이상, 프로젝트당 7~8명, 5개 프로젝트, 16개 제품, Main Release당 변경 30~40건기존 방식릴리즈 범위 확정 후 약 3일 동안 개발자가 SVN Merge 결과와 코드를 다시 읽어 문서를 수기 작성QA 문제알고리즘·공용 옵션·Trunk 변경 누락으로 의도된 변경이 Fake Issue로 재등록됨

해결 구조

1. PR 단위 변경 분석
PR 생성 시 Claude API 기반 Agent가 Commit, Jira, C++ 심볼, 기존 릴리즈 문서와 영향 범위를 분석했습니다.
2. 실제 WinApp 재현
Rust 기반 Custom MCP가 화면 정보를 CLI로 노출하고 버튼 클릭, TextBox 수정, 검사 Window 설정과 검사 흐름을 실행했습니다.
3. 문서·실행 증적 생성
변경 의도, 목적, 영향 범위, 제약사항을 Docmost에 기록하고 PR에 연결했습니다. 실제 Screenshot이 없으면 Mock 이미지가 남아 검토 누락을 드러내도록 설계했습니다.
4. QA 품질 게이트
재현되지 않으면 NEED CHECK로 개발자에게 반환하고, 문서와 실제 실행 결과가 다르거나 Smoke가 실패하면 Merge를 차단했습니다. 개발자가 변경 내용을 승인하고 QA가 릴리즈 착수를 승인했습니다.

운영 전환

처음부터 차단하지 않고 6개월간 Warning-only로 운영하며 오탐과 예외를 줄인 뒤 공식 결재를 거쳐 Blocking Gate로 전환했습니다.

자동화하지 않은 범위회귀 전용 장비를 확보할 수 없었고 Simulator도 CUDA 기반 3D Image Draw 경로를 재현하지 못했습니다. CPU 기반 기능과 재현 가능한 UI 흐름만 자동화하고 GPU·광학·실장비 의존 영역은 수동 QA로 남겼습니다.
릴리즈 직전 일괄 문서 작성 대신 PR 단위 Incremental Review를 도입해 변경 누락과 Fake Issue를 줄이고, DLL 구성 문제를 분기당 1~2회 Merge 전에 차단했습니다.
실제 실패 사례와 극복 과정 펼쳐보기

1. 왜 기존 방식이 계속 실패했나

  • 커밋 메시지는 대부분 “고객사 요청사항으로 수정” 한 줄이었고, 담당 개발자가 아니면 변경 의도를 알기 어려웠습니다.
  • 알고리즘과 공용 옵션은 직접 수정된 화면 외 다른 제품과 검사 결과에도 간접 영향을 줬습니다.
  • 고객사 특수 패치가 Main과 장기간 분리되며 사실상 커스텀 제품처럼 변했고 Merge Conflict가 반복됐습니다.
  • 템플릿, 교육, Jira, 코드 주석을 시도했지만 상세히 쓸수록 QA 이슈와 개발자 대응 업무가 늘어나는 구조라 최소 정보만 남기는 경향이 유지됐습니다.

2. 초기 자동화가 만든 새 문제

  • Agent가 수기 문서보다 약 4배 많은 내용을 생성해 개발자 검토 부담이 오히려 증가했습니다.
  • 내부 함수명을 그대로 문서에 쓰는 바람에 QA와 고객이 이해하기 어려운 설명이 생성됐습니다.
  • 초기부터 Merge를 차단하면 현업 반발과 오탐 비용이 커질 가능성이 높았습니다.
극복: 릴리즈 직전 일괄 검토를 PR별 신규 변경분 검토로 바꾸고, 심볼을 사용자 UI 명칭에 매핑했습니다. Gate는 6개월간 Warning으로 신뢰도를 쌓은 뒤 Blocking으로 전환했습니다.

3. 실제 누락 발견 사례

  • 공용 Trunk의 그림자 연산 처리 엔진이 마이너 업데이트됐지만 변경 담당자가 명확하지 않았고 Mock Screenshot이 그대로 남았습니다.
  • 추적 결과 알고리즘 팀장이 버그 수정 후 반영 사실을 잊고 있었던 것으로 확인됐습니다.
  • 변경 의도와 실제 영향 범위를 다시 확인한 뒤 승인해, 공용 알고리즘 변경이 문서에서 누락되는 것을 막았습니다.

4. 현업 반대는 어떻게 풀었나

  • “코드를 본 개발자가 직접 써야 정확하다”는 의견에는 AI 작성을 강제하지 않고 동일 양식에 수기로 작성할 수 있도록 허용했습니다.
  • 다만 실제 Screenshot, 실행 검증, Smoke Gate는 동일하게 적용해 작성 방식과 품질 통제를 분리했습니다.
  • 제약사항 공개를 꺼리는 개발자에게 QA 관점이 아니라 고객 셀프 해결과 개발자 전화 감소 관점으로 설득했습니다.
결과: 고객지원 전화가 하루 약 2건에서 이틀에 약 1건으로 줄어 기능 개발에 집중할 수 있다는 체감 효과가 수용의 결정적 계기가 됐습니다.

5. 운영 실패 시 대응

  • 시스템 장애는 Claude Code가 Runbook에 따라 1차 조치하고, 자동 복구 실패 시 담당자에게 Escalation하도록 구성했습니다.
  • 문서와 실행이 불일치하면 AI 설명을 신뢰하지 않고 Fail 처리했습니다.
  • 검증할 수 없는 GPU·광학·실장비 영역은 자동화 성공으로 포장하지 않고 수동 QA 범위로 명시했습니다.

QA Automation 이력서로 돌아가기 · 회사명과 내부 원문은 익명화했습니다.
