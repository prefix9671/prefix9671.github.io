이 사례의 핵심은 문서를 많이 만든 것이 아니라, AI 에이전트와 사람이 같은 기준으로 작업 맥락을 복원하도록 문서의 권한, 진입점, 업데이트 트리거, 호환성 입구를 설계한 것입니다.

기획, 개발, QA가 각자 다른 해석을 들고 움직이면 출시 속도는 느려집니다. 문서 운영 체계는 고객 니즈, PMF 가설, 개발 계획, 테스트 계획, 회귀 기준을 같은 작업 루프 안에서 공유하기 위한 기반입니다.

1-page Summary

Problem

자동화 작업은 코드, 화면 증거, 테스트 케이스, 실행 환경, 변경 이력이 분산되기 쉽고, 새 세션이나 다른 담당자가 맥락을 복원하는 데 시간이 듭니다.

Approach

활성 문서 인덱스, 문서 허브, 현재 상태, 구조 문서, 업데이트 정책, 프로그램별 상세 문서를 계층화했습니다.

Core Statement

AI를 출시 프로세스에 넣으려면 프롬프트보다 먼저 문맥 공급망이 필요합니다. 이 문서 구조는 각 에이전트가 "현재 기준은 무엇인가", "어디까지 완료됐는가", "어떤 문서를 함께 갱신해야 하는가"를 채팅 기록 없이 복원하게 만드는 운영 체계입니다.

Design Principle

얇은 루트

루트 가이드는 시작점만 제공하고 상세 정책은 활성 문서로 위임합니다.

명시적 권한

문서가 충돌할 때 어떤 계층을 우선할지 미리 정합니다.

상세 격리

프로그램별 화면과 검증 문서는 공통 정책을 오염시키지 않도록 분리합니다.

Agent Context Sharing

AI 에이전트 간 작업 맥락 공유 의도

채팅 기록이 사라져도 문서만 읽고 현재 작업 기준을 복원하게 만드는 handoff 구조

Context Restoration Flow

Start session
-> Read active document registry
-> Read current goal, architecture, and documentation workflow
-> Add common or program-specific docs only when task scope requires it
-> Implement or edit
-> Update affected active docs and compatibility entries
-> Record meaningful changes for the next agent

Portfolio Value

출시 프로세스 가속 관점의 가치

문서 체계는 관리용 산출물이 아니라 빠른 실험 조직을 위한 맥락 인프라입니다.

How It Supports Faster Experimentation

기획과 QA의 동시 출발

고객 니즈를 기능 설명으로만 남기지 않고, PMF 가설, 검증 질문, 실패 조건, 테스트 후보로 함께 정리할 수 있습니다.

개발과 테스트 계획 병렬화

구현 전후 문맥을 같은 문서 체계에 두면 개발 계획과 테스트 계획이 같은 입력에서 동시에 만들어집니다.

fixture 기반 빠른 확인

시안과 테스트 fixture를 먼저 만들면 조직 간 해석 차이와 놓친 기능을 더 이른 시점에 발견할 수 있습니다.

회귀 검증의 조직 자산화

회귀 테스트 초안과 변경 이력을 축적하면 다음 릴리즈의 QA 기준을 더 빠르게 복원할 수 있습니다.

Careful Claim

이 사례는 AI가 QA나 PM을 대체한다는 주장이 아닙니다. AI가 읽을 수 있는 문서 구조와 업데이트 정책을 먼저 만들고, 사람이 최종 판단하는 구조 안에서 기획, 개발, QA의 문맥 전환 비용을 줄였다는 점이 핵심입니다.

Public Boundary

  • 공개 페이지에는 구조, 의도, 정책만 설명합니다.
  • 비공개 저장소의 원문, 스크린샷, 로그, 내부 실행값, 고객 식별 정보는 포함하지 않습니다.
  • 상세 구조 원문은 중립화된 Markdown 초안으로만 유지합니다.
