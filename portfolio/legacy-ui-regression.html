Legacy UI Regression Automation

문서·보안·관측성 제약이 있는 레거시 WPF 환경에서 변경 의도와 UI 상태를 검증 가능한 출시 자산으로 전환한 사례

Name: 시우 조
Case: Release QA / Testability
Format: PAAR Evidence Pack

VOC와 영업 아이디어, 개발 문서, 코드 변경이 서로 다른 저장소와 시점에 분산되어 릴리즈 단계의 변경 맥락이 손실되는 문제가 있었습니다. 보안 정책상 민감 문서는 기존 DRM 문서 체계를 유지해야 했고, 레거시 DRM은 AI 에이전트가 직접 읽을 수 없었습니다.

전면적인 문서 체계 교체 대신, 상대적으로 위험도가 낮은 프론트엔드 변경부터 SVN 변경 의도 → AI 코드 리뷰 → 릴리즈 영향 범위 → 정적 상태전이 테스트 → UI 회귀 자동화로 연결했습니다. AI는 승인자나 자동 배포자가 아니라, 변경 의도와 검증 후보를 구조화하는 보조 계층으로 제한했습니다.

1-page Summary

Positioning

문서 접근 제약과 낮은 테스트 가능성을 이유로 자동화를 포기하지 않고, 허용된 범위 안에서 변경 추적성과 관측성을 단계적으로 확보한 Release QA 사례.

Operating Pattern

Change Intent → AI Review → Traceable Release Scope → Static State Tests → Anchored UI Automation → OCR/Data Validation → Human Review Checkpoint.

Measured Outcome

42%p → 6%p상태전이 관련 QA 패치 빌드 이슈 발생률
3일 → 1일전체 보고서 배포 전 Full Regression 소요시간
분기 2~3회P1 이상 기존 기능 영향 사전 검출

Core Statement

이 사례의 핵심은 “레거시 UI를 AI가 대신 클릭했다”가 아니다. 보안과 조직 정책을 우회하지 않으면서, 코드에 남은 변경 의도와 화면 상태를 검증 가능한 증거로 연결하고, 자동화가 읽을 수 없는 UI에 최소 관측 지점을 설계해 출시 전 회귀 검증을 반복 가능한 운영 체계로 만든 것이다.

Problem, Constraints, and Decision

기술 문제가 아니라 문맥 손실·보안 정책·책임 경계의 결합 문제로 정의

PAAR Overview

P · Problem

릴리즈 시점에 변경 맥락이 소실

VOC, 영업 아이디어, 개발 문서, 코드와 주석이 분산되어 있었다. 브랜치 병합 후에는 작성자가 구현 과정의 실제 변경점을 기억하기 어려웠고 릴리즈노트가 추상화됐다.

A · Approach

Source를 변경 증거의 기준으로 사용

문서에 반영되지 않은 사실은 source와 diff에 남아 있다는 전제 아래, 변경 의도와 실제 구현을 연결하는 추적 구조를 설계했다.

A · Action

허용 범위부터 점진 적용

전사 문서 체계 변경이 아닌, 노출 위험이 낮은 프론트엔드 코드부터 SVN 변경 의도, AI 리뷰, 릴리즈 범위, 정적 테스트를 연결했다.

R · Result

변경 추적성과 회귀 검증 확보

상태전이 결함을 조기에 줄이고, 관측이 어려운 WPF 화면을 앵커·상대좌표·OCR로 검증해 야간 전체 회귀를 운영했다.

Why Frontend First

전사 문서와 핵심 알고리즘을 한 번에 AI에 연결하는 대신, 상대적으로 외부 노출 위험이 낮고 기능 결과를 화면 상태로 확인할 수 있는 프론트엔드 변경부터 적용했다. 이는 보안 요구를 낮춘 것이 아니라, 승인 가능한 위험 범위에서 효과를 먼저 검증하기 위한 단계적 도입이었다.

Release Readiness in a Non-CI Environment

자동화된 CI/CD 승인 게이트는 존재하지 않았다. 따라서 본 사례에서의 통제 지점은 Jenkins 차단 규칙이나 자동 배포 승인 기능이 아니라, 변경 의도·정적 테스트 결과·회귀 증거를 묶어 담당자가 릴리즈 가능 여부를 검토할 수 있도록 만든 수동 Release Readiness Review였다.

QA는 배포를 시스템적으로 승인하거나 차단하지 않았다. 대신 P1 이상 영향, 상태 불일치, OCR 신뢰도 저하와 같은 위험 신호를 근거와 함께 제시하고, 기존 책임 체계의 담당자가 보류·재검증·배포 여부를 결정하도록 했다.

Outcome

변경 의도와 실제 source diff를 같은 릴리즈 단위로 묶고, 반복적으로 발생하던 상태전이 결함을 정적 검증 후보로 전환했다. 그 결과 QA 패치 빌드에서 확인되던 관련 이슈 발생률이 42%p에서 6%p로 감소했다.

수치는 특정 적용 범위와 관찰 기간의 운영 결과이며, 전 제품·전 조직에 일반화하지 않고 해당 워크플로의 효과 지표로 사용했다.

Control Principles

AI is a reviewer, not an approver

AI는 변경 영향과 누락 질문을 생성하지만, 배포 판단과 최종 테스트 범위 결정은 기존 책임 체계가 유지한다.

Evidence before decision

자동 승인 기능을 가정하지 않고, 변경 의도·화면·함수·Jira·테스트 증거가 검토 가능한 상태인지 먼저 확인한다.

Observed Automation Limits

갱신 후 Automation Tree 초기화

화면을 갱신하면 UI Automation 영역이 비거나 컨트롤 참조가 사라져 기존 selector를 재사용하기 어려웠다.

커스텀 체크박스 표현 손실

리스트가 의미 있는 항목 대신 “박스, 박스, 박스”처럼 노출되어 값과 선택 상태를 식별하기 어려웠다.

데이터가 이미지로 렌더링

DB 비즈니스 스키마를 알고 있어도 화면 데이터가 커스텀 렌더링 이미지여서 프로그램 내부 값으로 직접 읽을 수 없었다.

테스트 후커 미도입

구조 노출 우려로 별도 테스트 인터페이스가 제공되지 않아 외부에서 확인 가능한 신호만으로 검증해야 했다.

Minimal-invasive Anchor Strategy

1. 화면에는 노출되지 않는 인식용 Anchor Button 배치
2. Anchor를 기준으로 대상 컨트롤 상대좌표 계산
3. 갱신 전 선택값 / 초기값을 자동화 프로세스 메모리에 저장
4. 리본·그리드 갱신 실행
5. 동일 조건을 유지한 상태에서 대상 위치 재계산
6. 선택값이 초기 기대값과 일치하는지 조회
7. 성공 시 다음 환경으로 진행, 불일치 시 증거 저장 및 중단

OCR 이중 검증과 야간 Full Regression

패치 전후 화면·보고서 데이터 정합성을 자동 비교하고 불확실한 표본만 사람에게 에스컬레이션

End-to-end Validation Flow

Customer-specific Patch Build
→ Test Environment Restore
→ Report Entry / Query / Refresh / Re-refresh
→ Screenshot Capture
→ Open-source OCR
→ In-house AOI OCR Cross-check
→ Patch Before / After Data Consistency Comparison
→ PASS / FAIL Classification
→ Uncertain Sample Escalation (about 3%)
→ Human Review
→ Release Evidence Pack

Double-check Design

Independent OCR Paths

오픈소스 OCR과 자사 AOI 검사기 OCR을 함께 사용해 단일 엔진의 인식 오류가 최종 판정으로 바로 이어지지 않도록 했다.

Confidence-based Escalation

두 결과가 불일치하거나 신뢰도가 낮은 표본은 자동 확정하지 않고 약 3% 수준의 샘플 사유와 증거를 사람에게 전달했다.

Before / After Consistency

패치 전후 동일 조건에서 보고서 값, 선택 상태, 갱신 결과를 비교해 기능 추가가 기존 결과를 변경하지 않았는지 검증했다.

Evidence Retention

실행 단계, 조건, 스크린샷, OCR 결과, 판정 사유를 남겨 자동화 실패와 제품 결함을 구분할 수 있게 했다.

Operational Result

66% 감소전체 보고서 Full Regression: 3일에서 1일
야간 실행고객사별 커스텀 패치를 퇴근 후 자동 검증
P1 사전 검출분기별 약 2~3회 기존 기능 치명 영향 확인

Release Control Boundary

  • 자동화는 배포를 승인하지 않는다.
  • 자동화된 CI/CD 차단 게이트가 있다는 주장을 하지 않는다.
  • 조건 동일성이 깨지면 결과를 확정하지 않는다.
  • OCR 불일치와 낮은 신뢰도는 사람 검토로 전환한다.
  • P1 이상 영향이 발견되면 근거를 담당자에게 전달하고 수동 보류·재검증 결정을 요청한다.

Careful Claim

이 사례는 모든 레거시 WPF 화면을 완전 자동화했다는 주장이 아니다. 또한 자동화된 승인 게이트를 구축했다는 주장도 아니다. 제품 코드 변경과 테스트 후커 도입이 제한된 환경에서, 검증 가능한 화면과 보고서 경로부터 자동화하고 불확실한 결과는 사람에게 되돌려 수동 릴리즈 판단의 근거와 반복 가능성을 높였다는 사례다.
