# 독일계 글로벌 부품사 AOI–MES 통합 시뮬레이션 및 AI Native 지속개선 환경 구축

## 1면 압축판

기존 MES 시뮬레이터의 소켓 프로토콜 호환성 문제를 해결하는 대체 시스템을 개발하고, Loader·AOI 검사기·Repair Station을 연계해 실제 생산공정의 **80% 이상을 재현**했습니다.

- MES 연동 결함 **41건을 양산 배포 전 발견·개선**, 패치 후 일일 고객 불만 **2~3건 → 0건**
- 현장 반복 VOC **7~8종을 사내에서 재현**, 기술지원·개발 대응시간 **분기 약 40시간 절감**
- 현지법인 출장 **4~5회 → 설치·교육 1회**, 출장·유류·체재비 약 **400만 원 절감**
- 비개발자와 QA가 자연어로 개선사항을 전달하고 직접 검증할 수 있도록 Codex Agent를 개발환경에 연동
- 장비 화면의 전·후 상태 **2개 프레임**, 사용자가 그림판으로 표시한 관심 영역, 자연어 설명과 작업 의도를 함께 해석하는 멀티모달 현장 이슈 분석 흐름 구축
- Agent 변경사항을 GitLab Dev 브랜치에 반영하고 개발 MES에서 즉시 연동 검증한 뒤, 책임자·개발자가 SRP 준수, 중복 객체·컨테이너 생성, 변경 범위와 회귀 위험을 검토하여 Main에 병합

## 1. 문제 상황

기존 MES 시뮬레이터는 실제 현장 MES와 소켓 메시지 전처리 방식이 달랐습니다. 현장에서는 특정 접두·제어문자를 정규식으로 제거한 뒤 처리했지만, 제공된 시뮬레이터 실행 파일에는 같은 처리가 없어 정상 응답이 오지 않았습니다.

제공된 프로그램은 수정할 수 없는 EXE였고, 이를 그대로 사용하면 다음과 같은 실제 공정 핵심 상호작용을 사내에서 검증할 수 없었습니다.

- MES 동작용 바코드 입력
- 검사정보 검색
- PCB 생산라인 모델 변경
- 설비 상태와 MES 이벤트 간 순서 검증
- Loader·검사기·Repair Station 간 전·후단 상호작용

결과적으로 패치의 통합 문제는 사내가 아니라 현장에서 발견되었고, 각 공정에서 일일 2~3건의 불만이 발생하며 현지법인 출장이 반복됐습니다.

## 2. 대체 MES 시뮬레이터 구축

개발팀이 제공한 연동 코드 설명, 예시 데이터와 통신 규약을 기준으로 웹 서버와 소켓 통신용 텔레그램 요청·응답을 구현했습니다.

단순히 기존 EXE를 복제하는 것이 아니라, 실제 공정에서 필요한 상호작용을 독립적으로 재현할 수 있는 검증환경을 목표로 했습니다.

### 구현 범위

- MES 바코드 입력
- 검사정보 조회 및 응답
- PCB 생산라인 모델 변경
- 소켓 연결·요청·응답 및 메시지 프레이밍
- 공정 이벤트별 정상·예외 응답
- 테스트 데이터와 상태 전이 관리

이 환경을 통해 실제 MES가 연결되지 않은 상태에서도 검사장비 기능과 MES 연동 로직을 반복 검증할 수 있게 했습니다.

## 3. 실제 생산라인 수준으로 확장

기본 MES 상호작용이 동작한 뒤 Loader와 Repair Station을 각각 연계하여 단일 프로그램 테스트를 생산라인 흐름 검증으로 확장했습니다.

### 재현한 공정 흐름과 예외

- 보드 인입 시 Loader 대기
- 검사 진행 중 전단·후단 설비 순서 강제
- 검사 완료 후 Repair Station 이관
- Repair Station 가용 공간 부족
- Repair Plus와의 상호작용
- 모델 변경 전후의 설비·MES 상태 동기화
- 정상·재검사·대기·예외 상황의 이벤트 순서

이를 통해 실제 생산공정의 **80% 이상**과 현장에서 반복적으로 누적되던 주요 VOC **7~8종**을 사내에서 재현했습니다.

## 4. MES 결함 선제 발견과 운영 성과

새 검증환경에서 기존 테스트로는 발견하지 못했던 MES 연동 결함 **41건**을 양산 패치 배포 전에 추가 발견하고 개선했습니다.

### 정량 성과

- 패치 후 일일 고객 불만: **기존 2~3건 → 0건**
- 패치 대응 목적 현지법인 출장: **4~5회 → 설치·교육 1회**
- 실제 지급 기준 출장비·유류비·체재비: **약 400만 원 절감**
- 현장 기술지원 및 개발자의 반복 VOC 대응시간: **분기 약 40시간 절감**

핵심 변화는 현장에서만 발견되던 문제를 사내에서 반복 실행할 수 있는 테스트 입력으로 전환한 것입니다.

## 5. 비개발자·QA 참여형 Codex Agent 연동

시뮬레이터가 특정 개발자만 수정할 수 있는 도구가 되면 현장 VOC가 다시 대기열에 쌓이므로, 비개발자와 QA도 필요한 변경을 자연어로 설명하고 검증할 수 있도록 Codex Agent를 개발환경에 연동했습니다.

사용자는 코드 구조를 알지 못해도 다음 정보를 전달할 수 있게 했습니다.

- 어떤 업무를 수행하려 했는지
- 화면에서 무엇이 예상과 달랐는지
- 어떤 영역이 문제인지
- 정상적으로 기대한 결과가 무엇인지

Agent는 이를 기존 기능과 통신 흐름에 연결해 변경안을 생성하되, Main 브랜치나 운영환경에 직접 반영할 수 없도록 분리했습니다.

## 6. 장비 화면 2프레임 기반 멀티모달 이슈 이해

원격지 장비 문제는 한 장의 스크린샷만으로는 상태 변화와 사용자의 의도를 파악하기 어려웠습니다. 이를 보완하기 위해 장비 화면의 전·후 상태를 나타내는 **2개 프레임**을 입력으로 사용했습니다.

사용자는 화면을 그림판에서 열어 문제 위치에 동그라미·화살표·메모를 표시하고, 자연어로 다음 내용을 설명했습니다.

- 첫 번째 화면에서 수행한 작업
- 두 번째 화면에서 잘못된 상태
- 사용자가 기대한 정상 결과
- 해당 문제가 공정 흐름에서 왜 중요한지

멀티모달 Agent는 다음 정보를 함께 해석하도록 구성했습니다.

1. 변경 전 장비 화면 프레임
2. 변경 후 또는 오류 발생 화면 프레임
3. 사용자가 직접 표시한 관심 영역
4. 자연어 현상 설명
5. 사용자가 수행하려던 업무 의도

목적은 단순한 이미지 비교가 아니라, **화면 변화와 사용자의 작업 의도를 결합해 어떤 MES 신호·설비 상태·공정 순서가 잘못됐는지 추론할 수 있도록 돕는 것**이었습니다.

## 7. GitLab Dev–Review–Main 지속개선 흐름

Agent가 생성한 변경사항은 GitLab의 Dev 브랜치에만 푸시하도록 했습니다. 담당자는 개발 MES와 실제로 연동해 기능이 의도대로 동작하는지 바로 확인할 수 있었습니다.

### 반영 절차

1. 현장 또는 QA가 화면 2프레임과 자연어로 문제·의도 설명
2. Agent가 관련 코드와 통신 흐름 분석
3. 변경안을 Dev 브랜치에 반영
4. 개발 MES에서 실제 연동 확인
5. 담당자가 기능 결과 검증
6. 책임자와 개발자가 코드 구조 및 회귀 위험 리뷰
7. 승인된 변경만 Main 브랜치에 병합

### 코드리뷰 기준

- 단일 책임 원칙(SRP)을 유지하는가
- 기존 책임을 우회하기 위해 객체나 컨테이너를 중복 생성하지 않았는가
- 변경 범위가 요청 목적보다 과도하게 확대되지 않았는가
- 기존 상태관리와 객체 생명주기를 훼손하지 않는가
- 다른 공정·화면·장비 연동에 회귀를 유발하지 않는가
- 원복 가능한 크기로 변경이 격리되어 있는가

이 구조로 Agent의 구현 속도는 활용하되, 운영 반영 책임과 아키텍처 판단은 사람에게 남겼습니다.

## 8. 프로젝트의 AX 의미

이 프로젝트는 MES 시뮬레이터 한 개를 개발한 사례가 아니라 다음의 폐루프를 구축한 사례입니다.

> 현장 VOC 수집 → 화면·의도 멀티모달 분석 → Agent 변경 → Dev MES 실연동 검증 → 사람의 코드·위험 리뷰 → Main 반영

현장의 불만을 일회성 기술지원으로 처리하지 않고, 사내에서 재현하고 반복 검증하며 다음 배포에 반영할 수 있는 지속 가능한 개발환경으로 전환했습니다.

## 공개 범위

고객사명, 내부 제품명, 실제 장비 화면, 원문 통신 규약, 메시지 전문, 저장소 구조와 보안상 민감한 시스템 구성은 공개하지 않습니다.
