1-page Summary

Positioning

AI-native Workflow / QA Automation Lead. 모호한 개발·QA·운영 업무를 AI가 읽을 수 있는 입력, 통제 가능한 출력, 재사용 가능한 조직 자산으로 바꾸는 역할.

Operating Pattern

Problem → Input Sources → AI-assisted Review → Human Review → Test Gate → Reusable Asset. AI는 답변자가 아니라 검증 질문과 후보를 만드는 리뷰어로 제한.

Selected Proof Case

  • Why started: 릴리즈노트가 고객 설명 수준에 머물러 QA 검증 범위와 회귀 위험을 정의하기 어려웠다.
  • Input: 릴리즈노트, SVN diff, 함수 호출 관계, 기존 TC, UI 액션로그, 고객 사용 시나리오.
  • Approach: LLM을 릴리즈노트 작성자가 아니라 변경 영향도 리뷰어와 누락 질문 생성기로 사용했다.
  • What broke: 입력 맥락 부족 시 AI 출력이 일반론으로 흐르고, 고객용/내부용 산출물이 섞였으며, 출력 재현성이 흔들렸다.
  • Rejected: 릴리즈노트 단독 prompting, 매 릴리즈 전체 회귀 테스트, 개발팀 문서 강화만으로 해결하는 방식은 각각 구체성·비용·조직 의존성 문제로 제외했다.
  • Controls: 고객용/내부용 산출물 분리, 입력/출력 스키마 고정, Golden Test Set, Jenkins smoke/nightly regression 후보 분류를 적용했다.
  • Assetized: 릴리즈 QA 템플릿, 누락 질문 패턴, 회귀 후보 분류 기준, Golden Test Set으로 조직 자산화했다.

Core Statement

이 사례의 핵심은 “AI로 릴리즈노트를 예쁘게 작성했다”가 아니다. 추상적인 변경 설명을 AI가 읽을 수 있는 입력으로 구조화하고, 통제 가능한 초안과 사람 리뷰, Golden Test Set, Jenkins 후보군을 통해 반복 가능한 릴리즈 QA 자산으로 전환했다는 점이다.

PAAR Overview

P · Problem

릴리즈노트가 검증 기준이 되지 못함

“정렬 오류 수정”, “화면 갱신 개선” 같은 문장은 고객 설명으로는 충분할 수 있으나, QA 입장에서는 어느 화면, 어떤 조건, 어떤 회귀 영역을 봐야 하는지 알기 어려웠다.

A · Approach

AI를 변경 영향도 리뷰어로 사용

릴리즈노트만 AI에 넣지 않고 SVN diff, 기존 TC, UI 액션로그, 고객 사용 시나리오를 함께 입력했다. AI의 역할은 정답 생성이 아니라 누락 질문과 테스트 후보 생성으로 제한했다.

A · Action

스키마와 리뷰 게이트 설계

고객용/내부 QA용 산출물을 분리하고, 입력 스키마와 출력 포맷을 고정했다. AI 출력은 Golden Test Set, 기존 회귀 TC, 사람 리뷰를 통해 검증했다.

R · Result

반복 가능한 QA 자산화

릴리즈 QA 템플릿, 누락 질문 패턴, 회귀 후보 분류 기준, Golden Test Set, Jenkins smoke/nightly regression 후보군으로 조직 자산화했다.

Failures, Controls, and Assets

터진 지점, 제외한 대안, 봉합 방식, 조직 자산화 결과

What Broke

입력 맥락 부족 → 일반론 출력

릴리즈노트가 짧고 추상적이면 AI도 “정상 동작 확인” 수준의 뻔한 답을 생성했다.

SVN diff만으로는 업무 의미 부족

코드상 변경은 작은 함수 수정처럼 보여도 실제로는 UI 표시, export, 고객 리포트, 설정 저장, 고객별 옵션 동작에 영향을 줄 수 있었다.

기존 TC 품질에 AI가 종속

기존 테스트케이스가 허술하거나 오래되면 AI가 그 한계를 그대로 재생산했다.

고객용/내부용 산출물 혼재

고객용 릴리즈노트는 간결해야 하고, 내부 QA 체크리스트는 공격적으로 상세해야 한다. 두 목적을 한 문서로 처리하면 둘 다 품질이 떨어졌다.

Alternatives Rejected

AlternativeWhy Rejected
개발팀이 릴리즈노트를 더 상세히 작성작성자 개인 역량 의존, 개발팀 부담 증가, 고객용/내부 QA 기준 혼재 문제.
QA가 매 릴리즈마다 diff를 수동 분석정확도는 높을 수 있으나 확장성이 낮고 레거시 코드 지식 의존성이 커진다.
매 릴리즈 전체 회귀 테스트 수행장비, 시간, 환경, 데이터 초기화 비용이 커서 기본 전략으로 삼기 어렵다.
릴리즈노트만 AI에 입력입력 근거가 부족해 일반론적 체크리스트를 생성한다.
AI 출력을 바로 최종 산출물로 사용QA와 고객 커뮤니케이션에서 책임 문제가 생길 수 있다.

Fixes and Controls

Output Separation

고객용 릴리즈노트와 내부 QA 체크리스트를 분리했다. 고객용은 간결한 설명, 내부용은 조건/설정/export/report/회귀 위험까지 포함하는 검증 질문으로 구성했다.

Question-first AI Usage

AI에게 최종 답을 만들게 하지 않고, 누락 질문과 테스트 후보를 먼저 생성하게 했다.

Fixed Input / Output Schema

Release Note, Code Change, Existing TC, UI Context, Field Context를 고정 입력으로 만들고, 변경 요약, 영향 영역, 누락 질문, 테스트 후보, 회귀 위험을 고정 출력으로 만들었다.

Golden Test Set + Jenkins Gate

반복적으로 누락되는 질문은 Golden Test Set에 추가했다. 자동화 가능한 항목은 smoke 후보로, 장시간/환경 의존 항목은 nightly regression 후보로 분류했다.

Why This Matters

이 개선안의 목적은 릴리즈노트를 더 자세히 쓰게 하는 것이 아니다. 릴리즈노트를 UI, 함수, DB, 콜그래프, AST, 테스트케이스, 액션로그를 연결하는 변경 영향 추적 자산으로 발전시키는 것이다.

결과적으로 수동 QA는 일회성 확인 작업으로 끝나지 않고, 이후 UI regression automation과 self-healing의 학습 입력으로 재사용된다.

Self-healing Control Principle

Automation

기존 selector 실패 시 screen, label, nearby control, action intent, expected state를 기준으로 후보 control을 재탐색한다.

Human Review

self-healing 결과를 즉시 확정하지 않고, selector mapping 업데이트 후보를 생성한 뒤 사람 리뷰를 거쳐 rule을 갱신한다.

상세보기: AI-assisted Release QA

1.1 Why started

장비 소프트웨어 릴리즈 과정에서 가장 큰 문제는 릴리즈노트가 QA 검증 기준으로 사용하기에 충분히 구체적이지 않다는 점이었다.

개발팀 입장에서 릴리즈노트는 보통 “무엇을 수정했다”를 설명하는 문서였다. 예를 들어 “검사 결과 리스트 정렬 오류 수정”, “설정 저장 오류 개선”, “특정 조건에서 화면 갱신 오류 수정”과 같은 문장은 고객 전달용 설명으로는 어느 정도 의미가 있지만, QA 입장에서는 곧바로 테스트 가능한 기준이 되지 않았다.

QA가 실제로 확인해야 하는 질문은 훨씬 구체적이었다.

  • 어느 화면에서 발생하는 문제인가?
  • 어떤 입력 조건에서 재현되는가?
  • 정렬 기준은 무엇인가?
  • 필터 적용 후에도 동일하게 동작하는가?
  • 저장 후 재실행해도 유지되는가?
  • export/report 결과에도 영향이 있는가?
  • 고객사별 설정이나 장비 옵션에 따라 동작이 달라지는가?

즉 기존 릴리즈노트는 “설명문”이었고, QA에 필요한 것은 “검증 가능한 질문과 체크리스트”였다.

이 프로젝트는 릴리즈노트, 코드 변경, 기존 테스트케이스, UI 동작 로그, 고객 사용 시나리오가 서로 분리되어 있던 문제를 해결하기 위해 시작했다. 목표는 AI를 단순 문서 작성 도구로 쓰는 것이 아니라, 릴리즈 변경사항을 테스트 가능한 질문, 누락 위험, 회귀 테스트 후보, 고객용/내부용 산출물로 전환하는 것이었다.

1.3 Chosen approach

선택한 접근은 LLM을 “릴리즈노트 작성자”가 아니라 “변경 영향도 리뷰어”로 사용하는 것이었다.

단순히 릴리즈노트를 AI에게 넣고 문장을 다듬게 하는 방식은 배제했다. 그 방식은 문서 품질을 좋아 보이게 만들 수는 있지만, QA가 실제로 검증해야 하는 질문을 충분히 만들지 못했다.

대신 AI의 역할을 다음과 같이 제한했다.

  • AI는 최종 판단자가 아니라 초안 생성자다.
  • AI는 정답을 확정하지 않고, 사람이 놓칠 수 있는 질문을 생성한다.
  • AI는 고객용 문서와 내부 QA 문서를 분리해서 생성한다.
  • AI 출력은 Golden Test Set, 기존 회귀 테스트, 사람 리뷰를 거쳐야 한다.

이 접근의 핵심은 “AI가 답을 내는 것”이 아니라 “AI가 검증 질문을 생성하게 하는 것”이었다.

예를 들어 릴리즈노트에 “검사 결과 리스트 정렬 오류 수정”이라고 되어 있다면, AI가 단순히 “정렬이 정상 동작하는지 확인”이라고 답하는 것은 가치가 낮다. 실제로 필요한 것은 다음과 같은 질문이다.

- 어느 화면의 검사 결과 리스트인가?
- 기본 정렬 기준은 무엇인가?
- 새 검사 결과가 추가된 뒤에도 정렬이 유지되는가?
- 필터 적용 후 정렬이 유지되는가?
- 화면을 닫았다 다시 열어도 정렬 기준이 유지되는가?
- export/report 결과에도 동일한 정렬이 반영되는가?
- 고객별 옵션이나 장비 설정에 따라 정렬 기준이 달라지는가?

workflow는 다음과 같이 설계했다.

Release Note + SVN Diff + Existing TC + UI Context
→ AI-assisted Impact Review
→ Missing Questions
→ Test Candidates
→ Risk Summary
→ Internal QA Checklist
→ Customer-facing Release Note Draft
→ Human Review
→ Jenkins Smoke / Nightly Regression Candidate

1.4 What broke

초기 적용 과정에서는 여러 문제가 발생했다.

입력 맥락 부족

릴리즈노트가 “정렬 오류 수정”처럼 짧고 추상적이면 AI도 “정렬이 정상적으로 동작하는지 확인” 같은 뻔한 답을 생성했다. 이 경우 AI는 QA 업무를 도와주는 것이 아니라, 사람이 이미 알고 있는 말을 다시 정리하는 수준에 머물렀다.

SVN diff만으로는 업무 의미 부족

코드상으로는 특정 정렬 함수나 데이터 갱신 함수가 수정된 것처럼 보여도, 실제 영향은 UI 표시, export 결과, 고객 리포트, 설정 저장, 장비 옵션과 연결될 수 있었다. 코드 변경만 보고 영향 범위를 판단하면 과소평가가 발생할 수 있었다.

기존 TC 품질에 AI가 종속

기존 테스트케이스가 구체적이지 않거나 오래된 경우, AI는 그 한계를 그대로 따라갔다. 결과적으로 AI가 새로운 검증 관점을 제안하기보다 기존의 부족한 테스트 구조를 정중하게 재포장하는 문제가 있었다.

고객용/내부용 산출물 혼재

고객에게 전달할 릴리즈노트는 간결하고 안정적인 표현이 필요하지만, 내부 QA 체크리스트는 매우 구체적이고 공격적인 질문이 필요하다. 두 목적을 하나의 문서로 처리하려고 하면 고객용 문서는 지나치게 복잡해지고, 내부 QA 문서는 지나치게 약해졌다.

AI 출력 재현성

동일한 변경사항이라도 입력 방식이나 프롬프트가 조금 달라지면 어떤 실행에서는 export 검증 질문이 나오고, 다른 실행에서는 빠지는 식의 편차가 발생했다. QA 프로세스에서는 이런 편차가 위험했다.

1.5 Alternatives rejected

AlternativeWhy rejected
개발팀이 릴리즈노트를 더 상세히 작성문서 품질이 작성자 개인 역량에 의존하고, 개발팀 부담이 커진다. 고객용 설명과 내부 QA 기준은 목적이 달라 하나의 문서로 해결하기 어렵다.
QA가 매 릴리즈마다 diff를 수동 분석정확도는 높을 수 있으나 확장성이 낮고, 레거시 코드/모듈 지식에 대한 담당자 의존성이 커진다.
매 릴리즈 전체 회귀 테스트 수행장비, 시간, 테스트 환경, 데이터 초기화 비용이 커서 기본 전략으로 삼기 어렵다. 변경 영향 기반 선별이 필요하다.
릴리즈노트만 AI에 입력가장 빠르지만 입력 근거가 부족해 일반론적 체크리스트를 생성한다. 실제 변경 영향 분석이 되지 않는다.
AI 출력을 바로 최종 산출물로 사용QA와 고객 커뮤니케이션에서 책임 문제가 생길 수 있다. AI 출력은 반드시 사람 리뷰와 검증 게이트를 거쳐야 한다.

1.6 Fixes and controls

고객용/내부용 산출물 분리

고객용 릴리즈노트는 변경 내용을 간결하고 안정적으로 설명하는 데 집중했다. 내부 QA 체크리스트는 화면, 조건, 설정, 저장/불러오기, export/report, 회귀 위험까지 포함하는 공격적인 검증 질문으로 구성했다.

Question-first AI usage

AI에게 최종 답변보다 누락 질문을 먼저 생성하게 했다. AI가 바로 테스트케이스를 확정하면 그럴듯하지만 위험한 결과가 나올 수 있다. 반대로 AI가 “확인해야 할 질문”을 만들게 하면, 사람 리뷰어가 그 질문을 기준으로 실제 테스트 범위를 확정할 수 있다.

Golden Test Set 사용

특정 유형의 변경에서 반드시 나와야 하는 검증 질문을 기준으로 삼았다. 예를 들어 UI 정렬 변경이라면 필터, 새 데이터 추가, 재실행, export/report, 고객별 옵션 영향을 확인해야 한다. AI가 이런 질문을 반복적으로 누락하면 Golden Test Set에 추가해 다음 검토 때 비교 기준으로 사용했다.

Jenkins smoke/nightly regression 연결

AI가 생성한 테스트 후보 중 즉시 자동화 가능한 항목은 smoke test 후보로, 시간이 오래 걸리거나 환경 의존성이 있는 항목은 nightly regression 후보로 분류했다. 모든 것을 한 번에 자동화하려 하지 않고, 반복 가능성과 신뢰도가 높은 항목부터 검증 자산으로 전환했다.

1.7 Assetized lessons

이 프로젝트에서 얻은 가장 큰 교훈은 AI를 최종 판단자로 쓰면 위험하지만, 누락 질문 생성기와 검토 초안 생성기로 쓰면 실무 가치가 높다는 점이었다.

자산화 결과

  1. 릴리즈 QA 입력 템플릿

릴리즈노트를 그대로 검증하지 않고, 변경사항, 코드 diff, 기존 테스트케이스, UI 맥락, 고객 사용 시나리오를 함께 입력하는 구조를 만들었다.

  1. 고객용/내부용 산출물 분리 기준

고객용 문서는 안정적이고 간결해야 하며, 내부 QA 문서는 구체적이고 의심이 많아야 한다. 두 문서를 분리함으로써 고객 커뮤니케이션과 내부 검증 품질을 동시에 개선할 수 있었다.

  1. 누락 질문 패턴

정렬, 저장, 갱신, 필터, export, 권한, 네트워크, 재실행, 고객 옵션처럼 반복적으로 검증해야 하는 질문 패턴을 정리했다.

  1. Golden Test Set

AI가 자주 누락하는 질문이나 사람이 자주 놓치는 회귀 영역을 테스트셋으로 남겼다. 이를 통해 AI 출력의 품질을 감각이 아니라 비교 가능한 기준으로 확인할 수 있게 했다.

  1. Jenkins smoke/nightly regression 후보 분류 기준

모든 테스트를 한 번에 자동화하는 대신, 빠르게 반복할 smoke test와 장시간 검증이 필요한 nightly regression을 분리했다.

정리하면, 이 프로젝트의 결과물은 단순한 AI 프롬프트가 아니었다. 릴리즈 QA 업무를 반복 가능한 입력, 통제 가능한 AI 출력, 사람 리뷰, 자동화 후보, 조직 테스트 자산으로 전환한 workflow였다.

이 before/after의 핵심은 릴리즈노트 문장을 예쁘게 바꾸는 것이 아니다. 추상적인 변경 설명을 QA가 실행 가능한 질문과 체크리스트로 변환하고, 고객용 설명과 내부 검증 기준을 분리한 것이다.

1.9 Additional improvement plan: 릴리즈 QA 자산의 진화

이 workflow는 단순히 릴리즈노트를 AI로 요약하는 수준에서 끝나지 않는다. 목표는 릴리즈노트, UI 에셋, 함수, DB 연결, 콜그래프, AST Tree, 액션로그를 단계적으로 연결하여 “변경사항이 실제 사용자 화면과 테스트케이스까지 어떻게 이어지는지” 추적 가능한 릴리즈 QA 자산으로 발전시키는 것이다.

초기 단계: 개발자 직접 확인을 시스템화

초기 단계에서는 개발자가 릴리즈노트에 스크린샷을 직접 첨부하고, 변경된 화면과 동작을 스스로 확인하도록 시스템화한다.

기존에는 릴리즈노트가 텍스트 설명 중심이었기 때문에 QA가 변경 화면을 추측해야 했다. 이를 개선하기 위해 개발자가 수정한 UI, 변경 전/후 화면, 주요 설정값, 재현 조건을 릴리즈노트에 포함하도록 했다.

이 단계의 목적은 “개발자가 문서를 더 예쁘게 쓰게 하는 것”이 아니라, QA가 변경사항을 추측하지 않도록 최소한의 시각적 근거와 확인 지점을 남기는 것이다.

자산화 형태는 다음과 같다.

Developer Release Note Template
- 변경 요약
- 변경 화면 스크린샷
- 변경 전/후 비교
- 재현 조건
- 영향 가능 화면
- 개발자 직접 확인 항목
- QA 추가 확인 필요 항목

중기 단계: 릴리즈노트가 UI 에셋, 함수, DB 연결, 콜그래프의 암묵지를 포함

릴리즈노트가 누적되면 단순 변경 기록이 아니라 암묵지를 포함한 지식 자산이 된다.

예를 들어 특정 UI 화면이 어떤 함수와 연결되어 있는지, 그 함수가 어떤 DB 테이블 또는 설정값을 참조하는지, 어떤 고객 시나리오에서 자주 문제가 발생하는지에 대한 정보가 릴리즈노트와 함께 쌓인다.

이 단계에서는 릴리즈노트를 단순 문서로 보지 않고, UI 에셋, 함수, DB 연결, 콜그래프를 연결하는 메타데이터 레이어로 사용한다.

Release Note Metadata
- UI screen / component
- Related function
- Related DB table / query
- Related configuration
- Related customer scenario
- Related previous defect
- Related regression test

이렇게 누적된 릴리즈노트는 이후 AI-assisted impact review의 중요한 입력이 된다. AI는 더 이상 짧은 문장만 보는 것이 아니라, 과거 변경 이력과 UI-함수-DB 연결 맥락을 함께 참고할 수 있다.

후기 단계: AST Tree와 콜그래프로 알고리즘 변경과 UI 변경점을 연결

후기 단계에서는 AST Tree와 콜그래프를 사용해 알고리즘 변경과 UI 변경점을 묶을 수 있게 한다.

기존에는 알고리즘 변경과 UI 영향이 분리되어 보였다. 예를 들어 내부 검사 알고리즘, 정렬 로직, 데이터 집계 방식, 캐시 갱신 방식이 바뀌어도 실제로 어느 UI 화면, 어느 리포트, 어느 저장/로드 흐름에 영향을 주는지는 담당자 경험에 의존하는 경우가 많았다.

AST Tree와 콜그래프를 활용하면 변경된 함수가 어떤 상위 기능과 연결되는지, 어떤 UI 이벤트에서 호출되는지, 어떤 DB I/O 또는 리포트 출력으로 이어지는지 추적할 수 있다.

Algorithm Change
→ Changed Function
→ Call Graph
→ UI Event / Screen
→ DB Read / Write
→ Report / Export
→ Test Candidate

이 단계의 목표는 “코드가 바뀌었다”를 “사용자 화면과 테스트 후보가 바뀌었다”로 연결하는 것이다.

발전 단계: 표준화된 QA 양식 기반 TC와 Action Log 작성

다음 발전 단계는 표준화된 QA 양식에 따라 수동 테스터를 위한 테스트케이스와 액션로그를 작성하는 것이다.

이때 테스트케이스는 단순 해피패스만 포함하지 않는다. 장비 소프트웨어에서 반복적으로 문제가 되는 보편적인 예외 흐름을 포함해야 한다.

포함할 기본 흐름은 다음과 같다.

Standard QA Flow
- Happy path
- 갱신 / refresh
- 검사 / inspection
- 리뷰 / review
- 저장 / save
- 로드 / load
- 화면 재진입
- 프로그램 재실행
- 필터 적용 / 해제
- 설정 변경 후 복구
- export / report
- 네트워크 또는 DB 연결 예외

테스트케이스는 사람이 실행할 수 있어야 하고, 동시에 이후 자동화가 읽을 수 있는 action log로 남아야 한다.

Action Log Schema
- timestamp
- screen
- control
- action
- input value
- expected state
- actual state
- result
- screenshot reference
- related function / module

이 구조를 만들면 수동 테스트가 일회성 작업으로 끝나지 않는다. 사람이 수행한 테스트 흐름이 이후 UI regression automation의 입력 데이터가 된다.

구현 중: Action log 기반 UI regression automation과 self-healing

현재 구현 방향은 action log를 보는 UI regression automation과 self-healing이다.

기존 UI 자동화는 selector 변경, 화면 로딩 지연, 버튼 위치 변경, DevExpress/WPF control 구조 변경에 취약하다. 따라서 단순 selector 기반 자동화만으로는 장기 유지보수가 어렵다.

이를 해결하기 위해 action log를 기준으로 “사용자가 어떤 화면에서 어떤 의도로 어떤 동작을 했는지”를 기록하고, 자동화는 이 의도와 상태 변화를 기준으로 UI를 재탐색한다.

self-healing의 방향은 다음과 같다.

Self-healing UI Regression
1. 기존 selector로 control 탐색
2. 실패 시 screen / label / nearby control / action intent 기반 재탐색
3. 후보 control의 상태와 expected state 비교
4. 성공 시 selector mapping 업데이트 후보 생성
5. 사람 리뷰 후 selector rule 갱신

여기서 중요한 점은 self-healing을 완전 자동으로 확정하지 않는 것이다. UI 자동화의 self-healing은 잘못된 control을 선택할 위험이 있기 때문에, 자동 복구 후보를 만들고 사람 리뷰를 거쳐 rule을 갱신하는 방식이 더 안전하다.

추가 개선안의 핵심

이 개선안의 핵심은 릴리즈 QA를 다음 단계로 진화시키는 것이다.

텍스트 릴리즈노트
→ 스크린샷 포함 개발자 확인
→ UI / 함수 / DB / 콜그래프 메타데이터 누적
→ AST Tree + 콜그래프 기반 변경 영향 추적
→ 표준 QA 양식 기반 TC / Action Log 작성
→ Action Log 기반 UI regression automation
→ Human-reviewed self-healing

결과적으로 릴리즈노트는 단순 문서가 아니라, 코드 변경과 UI 동작, 테스트케이스, 자동화, 회귀 위험을 연결하는 조직 지식 자산으로 발전한다.
